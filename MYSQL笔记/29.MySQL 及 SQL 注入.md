MySQL 中的 SQL 注入（SQL Injection）是一种常见的网络攻击手段，攻击者通过在用户输入中插入恶意 SQL 代码，欺骗数据库执行非预期操作（如读取、修改、删除数据，甚至获取服务器权限）。理解 SQL 注入的原理和防御方法，对数据库安全至关重要。


### 一、SQL 注入的原理
SQL 注入的核心是**用户输入的数据被当作 SQL 代码的一部分执行**。当应用程序未对用户输入进行严格过滤时，攻击者可构造特殊输入，篡改原 SQL 语句的逻辑。

#### 示例：简单的登录场景
假设应用程序的登录验证 SQL 语句为：
```sql
-- 原逻辑：通过用户输入的 username 和 password 查询
SELECT * FROM users WHERE username = '用户输入的username' AND password = '用户输入的password';
```

若攻击者在 `username` 输入框中填入：  
`' OR '1'='1`  
则拼接后的 SQL 语句变为：
```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '用户输入的password';
```
由于 `'1'='1'` 恒为真，该语句会返回所有用户记录，攻击者无需正确密码即可登录。


### 二、常见的 SQL 注入攻击类型
1. **布尔盲注**：通过构造条件语句（如 `OR 1=1`、`OR 1=2`），根据页面返回结果（真/假）推断数据库信息。  
2. **时间盲注**：利用 `SLEEP()` 等函数，通过页面响应时间判断注入语句是否执行（如 `OR SLEEP(5)`）。  
3. **联合查询注入**：使用 `UNION SELECT` 拼接查询，获取其他表的数据（如 `' UNION SELECT 1,version(),3 -- `）。  
4. **堆叠查询注入**：通过分号 `;` 分隔多条 SQL 语句，执行删除、修改等操作（如 `' ; DROP TABLE users -- `）。  
5. **报错注入**：构造错误的 SQL 语句，从报错信息中提取数据库信息（如 `' AND (SELECT 1 FROM (SELECT COUNT(*),CONCAT(version(),FLOOR(RAND(0)*2))x FROM information_schema.tables GROUP BY x)a) -- `）。


### 三、SQL 注入的危害
- **数据泄露**：获取敏感信息（如用户密码、信用卡号）。  
- **数据篡改**：修改、删除数据库中的数据（如订单信息、用户权限）。  
- **权限提升**：通过 `GRANT` 等语句获取数据库管理员权限。  
- **服务器入侵**：若数据库权限过高，可能通过 `INTO OUTFILE` 写入恶意文件，控制服务器。  


### 四、防御 SQL 注入的核心方法
防御的关键是**将用户输入视为“数据”而非“代码”**，严格限制其对 SQL 语句的影响。

#### 1. 使用参数化查询（预编译语句）
这是最有效的防御方式。参数化查询将 SQL 语句结构与用户输入分离，用户输入仅作为参数传递，不会被解析为 SQL 代码。

**示例（以 Python + MySQL 为例）**：  
```python
# 错误方式：直接拼接字符串（易注入）
username = input("请输入用户名：")
sql = f"SELECT * FROM users WHERE username = '{username}'"  # 危险！

# 正确方式：使用参数化查询
import mysql.connector
conn = mysql.connector.connect(...)
cursor = conn.cursor(prepared=True)  # 启用预编译
sql = "SELECT * FROM users WHERE username = %s"  # %s 为参数占位符
cursor.execute(sql, (username,))  # 第二个参数为参数列表，自动转义
```

**MySQL 中的参数占位符**：不同语言驱动可能使用 `%s`、`?` 等，如 Java 的 `PreparedStatement` 使用 `?`。


#### 2. 输入验证与过滤
- **白名单验证**：只允许符合预期格式的输入（如用户名仅允许字母、数字、下划线，通过正则表达式校验）。  
  示例：`^[a-zA-Z0-9_]{3,16}$`（限制用户名长度 3-16，仅含字母、数字、下划线）。  
- **特殊字符转义**：对用户输入中的 SQL 特殊字符（如 `'`、`"`、`;`、`--`）进行转义。  
  MySQL 提供 `mysql_real_escape_string()` 函数（需注意字符集），但参数化查询更可靠。  


#### 3. 最小权限原则
- 数据库账号仅授予必要权限（如应用程序账号只给 `SELECT`、`INSERT` 权限，不给 `DROP`、`GRANT` 权限）。  
- 禁止使用 `root` 账号连接应用程序。  


#### 4. 避免暴露错误信息
- 生产环境中关闭数据库错误详情提示（如不向用户展示 `MySQL syntax error` 等信息），防止攻击者根据报错推断数据库结构。  
- 自定义错误页面，隐藏具体技术细节。  


#### 5. 其他辅助措施
- **使用 ORM 框架**：如 MyBatis、Hibernate 等，其内置的参数绑定机制可减少注入风险（但需正确使用，避免手动拼接 SQL）。  
- **定期安全审计**：使用工具（如 SQLMap）扫描应用程序，检测潜在注入点。  
- **限制数据库操作**：禁止使用 `EXECUTE`、`INTO OUTFILE` 等危险语句（通过数据库权限控制）。  


### 五、MySQL 特定的防护注意事项
- **避免使用 `SELECT *`**：减少敏感数据泄露风险。  
- **注意存储过程的安全**：存储过程中若拼接用户输入，同样可能导致注入，需使用参数化调用。  
- **升级 MySQL 版本**：新版本修复了部分安全漏洞（如限制 `堆叠查询` 在某些场景的使用）。  


### 总结
SQL 注入的本质是“输入未被正确处理而混入代码执行”，防御的核心是**参数化查询**，辅以输入验证、权限控制和错误隐藏。任何依赖用户输入构造 SQL 语句的场景，都必须严格遵循安全规范，从源头杜绝注入风险。
