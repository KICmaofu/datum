MySQL 索引是一种特殊的数据结构，用于快速查询数据库表中的特定记录。它类似于书籍的目录，能大幅提高查询效率，是数据库性能优化的核心手段。以下是关于 MySQL 索引的核心知识：


### 1. 索引的基本概念
- **作用**：减少数据库需要扫描的数据量，加快查询速度
- **代价**：索引会占用额外存储空间，且会降低插入、更新、删除操作的性能（需维护索引结构）
- **适用场景**：频繁查询的列、排序/分组的列、连接条件中的列


### 2. 索引的类型
MySQL 支持多种索引类型，适用于不同场景：

| 索引类型 | 说明 | 适用场景 |
|----------|------|----------|
| **B-Tree 索引** | 最常用的索引类型，基于 B+ 树结构 | 等值查询、范围查询、排序 |
| **Hash 索引** | 基于哈希表，查询速度极快 | 仅支持等值查询（=、IN），不支持范围查询 |
| **Full-Text 索引** | 全文索引，用于文本内容的关键词搜索 | 长文本字段（如文章内容）的模糊搜索 |
| **R-Tree 索引** | 空间索引，用于地理空间数据 | 存储经纬度等地理信息的字段 |
| **主键索引** | 特殊的唯一索引，不允许 NULL 值 | 表的主键列（自动创建） |
| **唯一索引** | 确保列值唯一（允许 NULL） | 需要唯一约束的列（如邮箱、手机号） |
| **复合索引** | 基于多个列创建的索引 | 多列组合查询（遵循最左前缀原则） |


### 3. 索引的创建与删除
#### ① 创建索引
```sql
-- 1. 创建表时定义索引
CREATE TABLE users (
  id INT PRIMARY KEY, -- 主键索引（自动创建）
  name VARCHAR(50),
  email VARCHAR(100) UNIQUE, -- 唯一索引
  age INT,
  INDEX idx_name (name), -- 普通索引
  INDEX idx_name_age (name, age) -- 复合索引
);

-- 2. 为已有表添加索引
ALTER TABLE users ADD INDEX idx_email (email);

-- 3. 创建全文索引
ALTER TABLE articles ADD FULLTEXT INDEX idx_content (content);
```

#### ② 删除索引
```sql
-- 方式1：ALTER TABLE
ALTER TABLE users DROP INDEX idx_name;

-- 方式2：DROP INDEX
DROP INDEX idx_age ON users;

-- 删除主键索引（需先移除 AUTO_INCREMENT）
ALTER TABLE users DROP PRIMARY KEY;
```


### 4. 索引的使用规则与优化
#### ① 索引生效的情况
- 等值查询（`=`、`IN`、`<=>`）
- 范围查询（`>`、`<`、`>=`、`<=`、`BETWEEN`）
- 前缀匹配（`LIKE 'prefix%'`，`%` 不在开头）
- 复合索引的最左前缀原则（使用索引的前 N 列）

```sql
-- 复合索引 (name, age) 生效的查询
SELECT * FROM users WHERE name = 'Alice'; -- 使用索引前1列
SELECT * FROM users WHERE name = 'Alice' AND age > 20; -- 使用索引全部列
SELECT * FROM users WHERE name LIKE 'A%' AND age = 25; -- 前缀匹配+第二列
```

#### ② 索引失效的情况
- 使用函数或表达式操作索引列（`WHERE YEAR(create_time) = 2023`）
- `LIKE` 以 `%` 开头（`WHERE name LIKE '%Alice'`）
- 隐式类型转换（字符串列用数字查询：`WHERE phone = 13800138000`）
- 使用 `NOT`、`!=`、`<>` 可能导致索引失效
- 复合索引不满足最左前缀原则（`WHERE age = 20`，跳过第一列 name）
- `OR` 连接的条件中，有一列无索引则整体失效


### 5. 复合索引的最左前缀原则
复合索引 `(col1, col2, col3)` 相当于创建了三个索引：
- `(col1)`
- `(col1, col2)`
- `(col1, col2, col3)`

**查询优化建议**：
- 将选择性高（区分度大）的列放在前面
- 避免创建冗余索引（如已有 `(a,b)`，无需再创建 `(a)`）
- 根据高频查询的字段顺序设计复合索引


### 6. 索引的优缺点
#### 优点
- 大幅提高查询速度（尤其是大表）
- 加速 `ORDER BY`、`GROUP BY` 操作（避免文件排序）
- 唯一索引可保证数据唯一性

#### 缺点
- 占用额外存储空间（可能达到表数据量的 10%-30%）
- 降低写操作（INSERT/UPDATE/DELETE）性能（需更新索引）
- 过多索引会增加查询优化器的选择负担


### 7. 索引使用的注意事项
1. **不要过度索引**：只为常用查询的列创建索引
2. **小表无需索引**：数据量少时，全表扫描可能比索引查询更快
3. **定期维护索引**：
   - 使用 `EXPLAIN` 分析索引使用情况
   - 重建碎片化索引（`ALTER TABLE table_name ENGINE = InnoDB;`）
4. **考虑索引选择性**：选择性低的列（如性别）不适合建索引
5. **避免索引列存 NULL**：NULL 值会影响索引效率，尽量用默认值


### 8. 索引分析工具：`EXPLAIN`
通过 `EXPLAIN` 可查看查询语句的执行计划，判断索引是否被有效使用：
```sql
EXPLAIN SELECT * FROM users WHERE name = 'Alice';
```

关键字段解读：
- `type`：访问类型（`const`、`eq_ref`、`ref` 表示索引有效；`ALL` 表示全表扫描）
- `key`：实际使用的索引名称（`NULL` 表示未使用索引）
- `rows`：预估扫描的行数（值越小越好）


索引是 MySQL 性能优化的关键，但并非越多越好。合理设计索引需要结合业务查询模式，平衡查询性能和写入性能，通过 `EXPLAIN` 工具持续分析和优化，才能发挥索引的最大价值。
