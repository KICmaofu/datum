MySQL 提供了丰富的数据类型，用于定义表中列存储的数据种类。选择合适的数据类型对数据库性能、存储空间和数据完整性至关重要。以下是主要数据类型分类及常用类型：


### **一、数值类型**
用于存储数字，分为整数和小数类型。

#### 1. 整数类型
| 类型         | 字节 | 范围（有符号）                          | 范围（无符号）                  | 用途示例               |
|--------------|------|-----------------------------------------|---------------------------------|------------------------|
| `TINYINT`    | 1    | -128 ~ 127                              | 0 ~ 255                         | 状态标记（0/1）、性别  |
| `SMALLINT`   | 2    | -32768 ~ 32767                          | 0 ~ 65535                       | 数量较少的ID           |
| `MEDIUMINT`  | 3    | -8388608 ~ 8388607                      | 0 ~ 16777215                    | 中等规模数据           |
| `INT`        | 4    | -2147483648 ~ 2147483647                | 0 ~ 4294967295                  | 常用ID、计数           |
| `BIGINT`     | 8    | -9223372036854775808 ~ 9223372036854775807 | 0 ~ 18446744073709551615       | 大整数（如订单号）     |

**示例**：
```sql
CREATE TABLE users (
  age TINYINT UNSIGNED,  -- 年龄（无符号，0~255）
  id INT PRIMARY KEY     -- 用户ID
);
```


#### 2. 小数类型
| 类型               | 说明                                  | 示例                          |
|--------------------|---------------------------------------|-------------------------------|
| `FLOAT(M,D)`       | 单精度浮点数，M总位数，D小数位数      | `FLOAT(5,2)` 表示 -999.99 ~ 999.99 |
| `DOUBLE(M,D)`      | 双精度浮点数，精度高于FLOAT           | `DOUBLE(10,3)`                |
| `DECIMAL(M,D)`     | 高精度小数（字符串存储），适合金额    | `DECIMAL(10,2)` 表示货币金额  |

**注意**：`DECIMAL` 精度最高，推荐用于财务数据；`FLOAT/DOUBLE` 适合科学计算。


### **二、字符串类型**
用于存储文本数据。

| 类型          | 长度限制               | 特点                                  | 用途示例               |
|---------------|------------------------|---------------------------------------|------------------------|
| `CHAR(M)`     | 0~255字符              | 固定长度，不足补空格                  | 手机号、身份证号       |
| `VARCHAR(M)`  | 0~65535字符            | 可变长度，按实际内容存储              | 用户名、地址           |
| `TEXT`        | 0~65535字符            | 长文本，适合大段文字                  | 文章内容、备注         |
| `MEDIUMTEXT`  | 0~16777215字符         | 中等长度文本                          | 长篇文章               |
| `LONGTEXT`    | 0~4294967295字符       | 极长文本                              | 日志、大型文档         |
| `ENUM('值1','值2')` | 单选枚举               | 只能存储枚举中的一个值                | 性别（'男','女'）      |
| `SET('值1','值2')`  | 多选集合               | 可存储多个值（用逗号分隔）            | 兴趣标签               |

**示例**：
```sql
CREATE TABLE articles (
  title VARCHAR(100) NOT NULL,  -- 文章标题（最多100字）
  content TEXT,                 -- 文章内容
  status ENUM('draft','publish') DEFAULT 'draft'  -- 状态
);
```


### **三、日期和时间类型**
用于存储日期和时间信息。

| 类型         | 格式                | 范围                                  | 用途示例               |
|--------------|---------------------|---------------------------------------|------------------------|
| `DATE`       | YYYY-MM-DD          | 1000-01-01 ~ 9999-12-31               | 生日、订单日期         |
| `TIME`       | HH:MM:SS            | -838:59:59 ~ 838:59:59                | 时长、时间段           |
| `DATETIME`   | YYYY-MM-DD HH:MM:SS | 1000-01-01 00:00:00 ~ 9999-12-31 23:59:59 | 注册时间、更新时间     |
| `TIMESTAMP`  | YYYY-MM-DD HH:MM:SS | 1970-01-01 00:00:01 ~ 2038-01-19 03:14:07 | 时间戳（自动更新）     |
| `YEAR`       | YYYY                | 1901 ~ 2155                           | 年份（如出版年份）     |

**特点**：
- `DATETIME` 存储实际时间，不受时区影响；
- `TIMESTAMP` 受时区影响，适合记录事件发生的绝对时间（如最后登录时间）。

**示例**：
```sql
CREATE TABLE orders (
  create_date DATE,               -- 订单日期
  create_time DATETIME,           -- 下单时间
  update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP  -- 自动更新时间
);
```


### **四、二进制类型**
用于存储二进制数据（如图片、文件）。

| 类型           | 长度限制               | 用途示例               |
|----------------|------------------------|------------------------|
| `BLOB`         | 0~65535字节            | 小型图片、文档         |
| `MEDIUMBLOB`   | 0~16777215字节         | 中等大小文件           |
| `LONGBLOB`     | 0~4294967295字节       | 大型文件               |

**注意**：实际开发中，通常不直接存储文件，而是存储文件路径，文件本身放在服务器磁盘。


### **五、选择数据类型的原则**
1. **最小够用**：如年龄用 `TINYINT` 而非 `INT`，节省空间；
2. **精确匹配**：如手机号用 `CHAR(11)`（固定长度），避免 `VARCHAR`；
3. **考虑扩展性**：如用户ID预计会增长到很大，用 `BIGINT` 而非 `INT`；
4. **语义化**：用 `DATE` 存储生日，而非字符串，便于日期计算。

合理选择数据类型是数据库优化的基础，直接影响查询效率和存储成本。
