MySQL 事务（Transaction）是一组不可分割的数据库操作序列，这些操作要么全部成功执行，要么全部失败回滚，确保数据的一致性和完整性。事务是处理并发操作和保证数据可靠性的核心机制。以下是关于 MySQL 事务的核心知识：


### 1. 事务的 ACID 特性
事务必须满足以下四个基本特性，简称 ACID：
- **原子性（Atomicity）**：事务中的所有操作要么全部完成，要么全部不完成（失败时回滚到初始状态）
- **一致性（Consistency）**：事务执行前后，数据库从一个一致状态转变为另一个一致状态（数据规则不被破坏）
- **隔离性（Isolation）**：多个事务并发执行时，彼此不会相互干扰（隔离级别不同，效果不同）
- **持久性（Durability）**：事务提交后，修改永久保存到数据库，即使系统崩溃也不会丢失


### 2. 事务的基本操作
MySQL 中默认是自动提交模式（每条 SQL 自动作为一个事务），可通过以下命令手动管理事务：

| 命令 | 描述 |
|------|------|
| `BEGIN` 或 `START TRANSACTION` | 开始一个新事务 |
| `COMMIT` | 提交事务，将所有修改永久保存 |
| `ROLLBACK` | 回滚事务，撤销所有未提交的修改 |
| `SAVEPOINT 保存点名称` | 在事务中创建保存点，用于部分回滚 |
| `ROLLBACK TO SAVEPOINT 保存点名称` | 回滚到指定保存点（不影响保存点之前的操作） |
| `SET AUTOCOMMIT = 0` | 关闭自动提交模式（需手动 `COMMIT`） |
| `SET AUTOCOMMIT = 1` | 开启自动提交模式（默认） |


### 3. 基本示例
假设 `accounts` 表记录用户账户余额：

| id  | name   | balance |
|-----|--------|---------|
| 1   | Alice  | 1000    |
| 2   | Bob    | 1000    |

实现 Alice 向 Bob 转账 200 元的事务：
```sql
-- 开始事务
BEGIN;

-- 步骤1：Alice 账户减少 200
UPDATE accounts SET balance = balance - 200 WHERE id = 1;

-- 步骤2：Bob 账户增加 200
UPDATE accounts SET balance = balance + 200 WHERE id = 2;

-- 检查操作是否正确（可选）
SELECT * FROM accounts WHERE id IN (1, 2);

-- 确认无误后提交（修改生效）
COMMIT;

-- 如果出错，回滚（所有修改撤销）
-- ROLLBACK;
```

**部分回滚示例**（使用保存点）：
```sql
BEGIN;
UPDATE accounts SET balance = balance - 200 WHERE id = 1;
SAVEPOINT after_alice; -- 创建保存点

UPDATE accounts SET balance = balance + 200 WHERE id = 2;
-- 发现 Bob 的操作有误，回滚到保存点（仅撤销 Bob 的操作）
ROLLBACK TO SAVEPOINT after_alice;

-- 重新执行 Bob 的操作
UPDATE accounts SET balance = balance + 200 WHERE id = 2;
COMMIT;
```


### 4. 事务隔离级别
当多个事务同时操作同一批数据时，可能产生脏读、不可重复读、幻读等问题。MySQL 提供了四种隔离级别解决这些问题：

| 隔离级别 | 说明 | 可能出现的问题 |
|----------|------|----------------|
| **READ UNCOMMITTED**（读未提交） | 允许读取未提交的事务修改 | 脏读、不可重复读、幻读 |
| **READ COMMITTED**（读已提交） | 只能读取已提交的事务修改 | 不可重复读、幻读 |
| **REPEATABLE READ**（可重复读） | 保证同一事务中多次读取结果一致 | 幻读（MySQL 已通过 MVCC 解决） |
| **SERIALIZABLE**（串行化） | 事务串行执行，完全隔离 | 无，但性能最差 |

**MySQL 默认隔离级别**：`REPEATABLE READ`（可重复读）

**查看和设置隔离级别**：
```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 临时设置隔离级别（当前会话有效）
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 全局设置（需重启生效）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```


### 5. 事务并发问题
- **脏读**：一个事务读取到另一个未提交事务的修改（可能被回滚）
- **不可重复读**：同一事务中，两次读取同一数据结果不同（被其他事务修改并提交）
- **幻读**：同一事务中，两次查询结果集行数不同（被其他事务插入/删除并提交）


### 6. 注意事项
1. **存储引擎支持**：只有 `InnoDB` 存储引擎支持事务，`MyISAM` 不支持
2. **锁机制**：事务执行时可能产生锁（行锁、表锁），长时间未提交的事务会导致锁等待
3. **事务大小**：避免超大事务（如批量处理10万行），会占用大量资源并导致锁竞争
4. **错误处理**：事务中若有 SQL 错误，需手动执行 `ROLLBACK`（部分错误会自动回滚）
5. **日志依赖**：事务持久性依赖于 redo log（重做日志），崩溃恢复时通过日志恢复数据


### 7. 适用场景
- 转账、支付等涉及多步数据修改的操作
- 订单创建（需同时修改库存、订单表、日志表等）
- 任何需要保证数据一致性的多步操作


事务是数据库保证数据完整性的核心机制，尤其在并发环境下至关重要。合理设置隔离级别、控制事务大小、及时提交或回滚，是保证系统性能和数据可靠性的关键。
