在 Django 中，视图（View）是处理用户请求并返回响应的核心组件，主要分为两种实现方式：**函数视图（Function-Based View, FBV）** 和 **类视图（Class-Based View, CBV）**。两者本质上都是处理请求的入口，但设计思想和使用场景不同，各有优劣。


### 一、函数视图（FBV）  
函数视图是最直观的视图形式，通过 Python 函数定义，接收 `request` 对象作为参数，根据请求方法（如 `GET`/`POST`）处理逻辑并返回响应。  


#### 1. 基本结构  
```python
# myapp/views.py
from django.http import HttpResponse, JsonResponse
from django.shortcuts import render

def my_view(request):
    # request 是 HttpRequest 对象，包含请求的所有信息（方法、参数、用户等）
    if request.method == "GET":
        # 处理 GET 请求（如显示表单、查询数据）
        return render(request, "myapp/form.html")  # 渲染模板
    elif request.method == "POST":
        # 处理 POST 请求（如提交表单、保存数据）
        data = {"status": "success", "message": "提交成功"}
        return JsonResponse(data)  # 返回 JSON
    else:
        # 处理其他请求方法（如 PUT、DELETE）
        return HttpResponse("不支持的请求方法", status=405)
```  


#### 2. 核心特点  
- **直观简单**：用函数封装逻辑，符合初学者对“请求-处理-响应”流程的理解。  
- **灵活可控**：可自由定义处理逻辑，不受类继承的约束。  
- **按请求方法分支**：通过 `if-elif` 判断 `request.method`，区分 `GET`/`POST` 等请求。  


#### 3. 适用场景  
- 简单的页面或接口（如静态页面、单一功能的表单提交）。  
- 逻辑简单、无需复用的场景（如一次性的特殊处理）。  


#### 4. 示例：处理表单提交（FBV）  
```python
from django.shortcuts import render, redirect
from .forms import LoginForm

def login(request):
    if request.method == "POST":
        # 绑定 POST 数据到表单
        form = LoginForm(request.POST)
        if form.is_valid():
            # 验证成功：执行登录逻辑
            username = form.cleaned_data["username"]
            # ... 验证密码、创建会话等 ...
            return redirect("home")  # 跳转首页
    else:
        # GET 请求：显示空表单
        form = LoginForm()
    
    # 渲染模板（GET 请求或验证失败的 POST 请求均走此逻辑）
    return render(request, "myapp/login.html", {"form": form})
```  


### 二、类视图（CBV）  
类视图是基于面向对象（OOP）思想的视图形式，通过类定义视图，继承自 Django 内置的 `View` 类（或其子类），并通过类方法（如 `get()`/`post()`）对应不同的请求方法。  


#### 1. 基本结构  
```python
# myapp/views.py
from django.http import HttpResponse, JsonResponse
from django.views import View

class MyView(View):
    # 处理 GET 请求
    def get(self, request):
        return render(request, "myapp/form.html")
    
    # 处理 POST 请求
    def post(self, request):
        data = {"status": "success", "message": "提交成功"}
        return JsonResponse(data)
    
    # 处理其他请求方法（如 PUT）
    def put(self, request):
        return HttpResponse("处理 PUT 请求", status=200)
```  


#### 2. 与 URL 关联  
类视图需通过 `as_view()` 方法转换为可调用的视图函数，才能在 `urls.py` 中配置：  
```python
# myapp/urls.py
from django.urls import path
from .views import MyView

urlpatterns = [
    path('my-view/', MyView.as_view(), name='my_view'),  # 必须调用 as_view()
]
```  


#### 3. 核心特点  
- **面向对象**：通过类的继承、方法重写实现代码复用（如通用功能抽离为父类）。  
- **请求方法分离**：不同请求方法（`GET`/`POST`）对应类中的不同方法（`get()`/`post()`），无需 `if-elif` 判断，结构更清晰。  
- **内置通用视图**：Django 提供大量通用类视图（如 `ListView`/`DetailView`），封装了常见逻辑（如列表查询、详情展示），减少重复代码。  


#### 4. 常用通用类视图  
Django 的 `django.views.generic` 模块提供了多个通用类视图，覆盖大部分常见场景：  

| 类视图          | 作用                          | 核心属性/方法                          |  
|-----------------|-------------------------------|---------------------------------------|  
| `View`          | 基础类视图，所有类视图的父类  | `get()`/`post()`/`dispatch()` 等      |  
| `TemplateView`  | 渲染指定模板（无复杂逻辑）    | `template_name`（模板路径）           |  
| `ListView`      | 展示模型列表（如文章列表）    | `model`（模型）、`queryset`（查询集）、`paginate_by`（分页） |  
| `DetailView`    | 展示模型详情（如文章详情）    | `model`、`pk_url_kwarg`（主键参数名） |  
| `FormView`      | 处理表单（如登录、注册）      | `form_class`（表单类）、`success_url`（提交成功跳转地址） |  


#### 5. 示例：通用类视图（`ListView` 和 `DetailView`）  
```python
# 展示文章列表
from django.views.generic import ListView
from .models import Article

class ArticleListView(ListView):
    model = Article  # 关联模型
    template_name = "myapp/article_list.html"  # 模板路径
    context_object_name = "articles"  # 模板中使用的变量名（默认 object_list）
    paginate_by = 10  # 每页显示 10 条

    # 自定义查询集（如只显示已发布的文章）
    def get_queryset(self):
        return Article.objects.filter(is_published=True).order_by("-create_time")


# 展示文章详情
from django.views.generic import DetailView

class ArticleDetailView(DetailView):
    model = Article
    template_name = "myapp/article_detail.html"
    context_object_name = "article"  # 模板中变量名（默认 object）
```  

对应的 URL 配置：  
```python
urlpatterns = [
    path('articles/', ArticleListView.as_view(), name='article_list'),
    path('articles/<int:pk>/', ArticleDetailView.as_view(), name='article_detail'),
]
```  


#### 6. 类视图中的装饰器  
FBV 可直接使用函数装饰器（如 `@login_required`），但 CBV 需通过 `@method_decorator` 将装饰器应用到类方法：  

```python
from django.utils.decorators import method_decorator
from django.contrib.auth.decorators import login_required

# 方式 1：装饰单个方法
class MyView(View):
    @method_decorator(login_required)  # 仅 get() 方法需要登录
    def get(self, request):
        return HttpResponse("需要登录才能访问")

    def post(self, request):
        return HttpResponse("POST 无需登录")

# 方式 2：装饰整个类（所有方法）
@method_decorator(login_required, name='dispatch')  # dispatch 是请求入口方法
class MyView(View):
    def get(self, request):
        return HttpResponse("整个类都需要登录")
```  


### 三、FBV 与 CBV 的对比  
| 维度         | 函数视图（FBV）                          | 类视图（CBV）                          |  
|--------------|------------------------------------------|----------------------------------------|  
| **代码结构** | 单一函数，通过 `if-elif` 区分请求方法     | 类+方法，不同请求方法对应不同函数       |  
| **复用性**   | 较差，需通过函数调用复用逻辑             | 强，可通过继承、混入（Mixin）复用代码   |  
| **学习成本** | 低，符合函数式编程直觉                   | 高，需理解面向对象、继承、通用视图设计   |  
| **灵活性**   | 高，可自由编写逻辑，不受类约束           | 中等，需遵循类视图的继承和方法规范       |  
| **适用场景** | 简单逻辑（如静态页、单一表单）           | 复杂逻辑（如 CRUD 操作、分页、权限控制） |  


### 四、如何选择？  
- **优先用 FBV**：逻辑简单、无需复用、快速开发（如简单的接口、静态页面）。  
- **优先用 CBV**：逻辑复杂、需要复用（如多个视图共用分页/权限逻辑）、或使用通用视图简化开发（如列表/详情页）。  


### 总结  
FBV 和 CBV 是 Django 视图的两种实现方式，本质上都是处理请求的入口：  
- FBV 直观简单，适合简单场景，用函数封装逻辑；  
- CBV 基于面向对象，复用性强，适合复杂场景，尤其通过通用视图可大幅减少代码；  
- 实际开发中可根据需求混合使用（如简单接口用 FBV，复杂列表页用 CBV），核心是让代码更清晰、易维护。
