# Python logging 模块完全指南：日志记录与实战应用
logging 是 Python 内置的 **日志记录标准库**，核心功能是提供分级、可配置、多目的地的日志输出能力——支持将日志打印到控制台、写入文件、轮转切割大日志，还能自定义日志格式、过滤规则，兼具灵活性和可靠性。

与 `print` 语句相比，logging 支持日志分级（调试/信息/警告/错误）、无需手动删除调试代码、可持久化存储；与第三方日志库（如 `loguru`）相比，它无需额外安装，原生集成 Python 生态，支持复杂场景配置（如多模块日志、日志轮转、分布式日志），是 Python 项目（从脚本到大型应用）的首选日志解决方案。


## 一、核心定位与优势
### 1. 核心特点
- **内置无依赖**：Python 标准库自带，无需额外安装，导入即使用；
- **日志分级**：支持 5 级日志（DEBUG/INFO/WARNING/ERROR/CRITICAL），可按需过滤不同重要程度的日志；
- **多目的地输出**：同时输出到控制台、文件、网络服务（如 ELK）等，支持多个输出目的地组合；
- **灵活格式化**：自定义日志格式（包含时间、模块、行号、日志级别等信息），便于调试和分析；
- **日志轮转**：自动切割大日志文件（按大小/时间），避免单个日志文件过大，便于归档管理；
- **线程/进程安全**：多线程、多进程环境下日志记录不会错乱，无需手动加锁；
- **可配置性**：支持通过代码、配置文件、环境变量配置，适配开发/测试/生产等不同环境；
- **多模块支持**：不同模块可共享日志配置，统一日志风格，便于追踪跨模块问题。

### 2. 适用场景
- **项目调试**：输出变量值、函数调用流程，通过 DEBUG 级日志定位问题；
- **生产环境监控**：记录系统运行状态（INFO）、警告（WARNING）、错误（ERROR），便于问题回溯；
- **异常追踪**：记录异常堆栈信息，快速定位崩溃原因；
- **性能分析**：记录关键操作耗时，分析系统瓶颈；
- **安全审计**：记录用户操作、敏感接口调用，满足合规要求；
- **分布式系统日志**：配合日志收集工具（如 ELK、Fluentd），集中管理多实例日志。

### 3. 与 `print` 的对比（为什么不用 `print` 做日志？）
| 特性                | `print` 语句                          | `logging` 模块                          |
|---------------------|---------------------------------------|-----------------------------------------|
| 分级能力            | 无（所有输出同等重要）                | 支持 5 级分级，可过滤不重要日志          |
| 输出目的地          | 仅控制台                              | 控制台、文件、网络等多目的地            |
| 格式控制            | 需手动拼接时间、模块等信息            | 自动格式化，支持自定义模板              |
| 持久化存储          | 无（需手动写入文件）                  | 直接写入文件，支持日志轮转              |
| 多环境适配          | 需手动注释/删除调试输出               | 通过配置切换日志级别，无需修改代码      |
| 线程安全            | 非线程安全（多线程输出可能错乱）      | 线程/进程安全，适合并发场景            |
| 异常追踪            | 需手动打印 `traceback`                | 自动记录异常堆栈信息（`exc_info=True`） |

**结论**：小脚本临时调试可用 `print`，正规项目（尤其是长期维护、多模块、生产环境运行的项目）必须使用 `logging`。


## 二、环境准备
logging 是 Python 内置模块，**无需额外安装**，直接导入即可使用：
```python
import logging
```

### 版本说明
- Python 2/3 均支持 logging 模块，核心接口一致；
- Python 3.2+ 新增 `logging.config.dictConfig` 等高级配置接口，推荐使用 Python 3.6+ 以获得更完善的功能支持。

验证模块可用性（基础日志输出）：
```python
# 基础配置：输出到控制台，日志级别 WARNING 及以上
logging.basicConfig(level=logging.WARNING)

# 输出不同级别日志
logging.debug("调试信息（默认不输出）")
logging.info("普通信息（默认不输出）")
logging.warning("警告信息（默认输出）")
logging.error("错误信息（默认输出）")
logging.critical("严重错误信息（默认输出）")
```

输出结果：
```

ERROR:root:错误信息（默认输出）
CRITICAL:root:严重错误信息（默认输出）
```


## 三、核心概念：理解 logging 模块的架构
logging 模块采用 **“日志器-处理器-格式化器-过滤器”** 的分层架构，各组件职责明确，灵活组合：

### 1. 核心组件
| 组件          | 中文名称          | 核心职责                                  | 关键类/函数                          |
|---------------|-------------------|-------------------------------------------|-------------------------------------|
| **Logger**    | 日志器（入口）    | 发起日志记录请求（如 `logger.info()`），管理 Handler | `logging.getLogger(name)`           |
| **Handler**   | 处理器（输出）    | 将日志发送到指定目的地（控制台/文件/网络） | `StreamHandler`（控制台）、`FileHandler`（文件）等 |
| **Formatter** | 格式化器（格式）  | 定义日志输出格式（如时间、模块、日志内容） | `logging.Formatter(fmt, datefmt)`   |
| **Filter**    | 过滤器（过滤）    | 按规则过滤日志（如仅保留特定模块的日志）  | `logging.Filter(name)`              |

### 2. 组件关系
1. 一个 **Logger** 可以关联多个 **Handler**（如同时输出到控制台和文件）；
2. 一个 **Handler** 只能关联一个 **Formatter**（一个处理器对应一种格式）；
3. 一个 **Handler** 可以关联多个 **Filter**（多重过滤）；
4. Logger 有层级关系（如 `loggerA` 和 `loggerA.sub`），子 Logger 会继承父 Logger 的配置。

### 3. 日志级别（从低到高）
logging 定义 5 个标准级别，级别越高，日志越重要，默认只输出 **WARNING 及以上** 级别：
| 级别常量          | 数值 | 描述                                  | 适用场景                          |
|-------------------|------|---------------------------------------|-----------------------------------|
| `logging.DEBUG`   | 10   | 调试信息（如变量值、函数调用流程）    | 开发环境调试                      |
| `logging.INFO`    | 20   | 普通信息（如系统启动完成、用户操作）  | 记录正常运行状态                  |
| `logging.WARNING` | 30   | 警告信息（如参数不合法、资源不足）    | 潜在问题，不影响当前运行          |
| `logging.ERROR`   | 40   | 错误信息（如功能执行失败、异常抛出）  | 功能异常，需关注                  |
| `logging.CRITICAL`| 50   | 严重错误（如系统崩溃、数据库连接失败）| 致命问题，系统无法正常运行        |

### 4. 关键流程
1. 调用 `logger.debug()`/`info()` 等方法发起日志请求；
2. Logger 检查日志级别是否满足（如级别 >= Logger 设置的阈值）；
3. 满足则通过 Filter 过滤（不满足则丢弃）；
4. 过滤后的日志传递给关联的 Handler；
5. Handler 通过 Formatter 格式化日志；
6. Handler 将格式化后的日志输出到指定目的地。


## 四、核心功能：分场景详解（语法+示例）
### 1. 基础用法：快速配置（`logging.basicConfig()`）
`basicConfig()` 是 logging 高层接口，快速配置日志（默认创建 root Logger 和 StreamHandler），适合简单场景：

#### 语法
```python
logging.basicConfig(
    level=logging.WARNING,  # 日志级别（默认 WARNING）
    format="%(asctime)s - %(name)s - %(levelname)s - %(message)s",  # 日志格式
    datefmt="%Y-%m-%d %H:%M:%S",  # 时间格式
    filename="app.log",  # 输出到文件（不指定则输出到控制台）
    filemode="w",  # 文件打开模式（w：覆盖，a：追加，默认 a）
    encoding="utf-8"  # 文件编码（Python 3.9+ 支持）
)
```

#### 示例 1：输出到控制台（自定义格式）
```python
import logging

# 基础配置：输出到控制台，INFO 级别及以上，自定义格式
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s - %(levelname)s - %(module)s:%(lineno)d - %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S"
)

# 输出日志
logging.info("系统启动成功")
logging.warning("内存使用率超过 80%")
logging.error("数据库查询失败")

# 输出变量值（DEBUG 级别，需将 level 设为 DEBUG 才会显示）
name = "Python"
logging.debug(f"变量 name 的值：{name}")
```

输出结果（控制台）：
```
2025-11-20 15:30:00 - INFO - __main__:10 - 系统启动成功
2025-11-20 15:30:00 - WARNING - __main__:11 - 内存使用率超过 80%
2025-11-20 15:30:00 - ERROR - __main__:12 - 数据库查询失败
```

#### 示例 2：输出到文件（追加模式）
```python
import logging

# 配置：输出到 app.log 文件，WARNING 级别及以上，追加模式
logging.basicConfig(
    level=logging.WARNING,
    format="%(asctime)s - %(levelname)s - %(message)s",
    filename="app.log",
    filemode="a",  # 追加模式（默认），避免覆盖历史日志
    encoding="utf-8"
)

logging.warning("用户登录失败：密码错误")
logging.error("接口调用失败：超时")
```

执行后，`app.log` 文件内容：
```
2025-11-20 15:35:00,123 - WARNING - 用户登录失败：密码错误
2025-11-20 15:35:00,124 - ERROR - 接口调用失败：超时
```

### 2. 进阶用法：自定义 Logger/Handler/Formatter
`basicConfig()` 仅适用于简单场景，复杂场景（如同时输出到控制台和文件、不同级别日志输出到不同文件）需手动创建 Logger、Handler、Formatter：

#### 示例：同时输出到控制台和文件（不同格式）
```python
import logging

# 1. 创建 Logger（指定名称，避免使用 root Logger）
logger = logging.getLogger("my_app")
logger.setLevel(logging.DEBUG)  # Logger 级别（需低于 Handler 级别才会传递）

# 2. 创建 Formatter（两种格式：控制台简洁，文件详细）
# 控制台格式（简洁）
console_fmt = logging.Formatter("%(levelname)s - %(message)s")
# 文件格式（详细，包含时间、模块、行号）
file_fmt = logging.Formatter(
    "%(asctime)s - %(name)s - %(levelname)s - %(module)s:%(lineno)d - %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S"
)

# 3. 创建 Handler（控制台 + 文件）
# 控制台 Handler
console_handler = logging.StreamHandler()
console_handler.setLevel(logging.INFO)  # 控制台仅输出 INFO 及以上
console_handler.setFormatter(console_fmt)

# 文件 Handler（输出到 app.log）
file_handler = logging.FileHandler("app.log", encoding="utf-8")
file_handler.setLevel(logging.DEBUG)  # 文件输出 DEBUG 及以上（更详细）
file_handler.setFormatter(file_fmt)

# 4. 将 Handler 绑定到 Logger
logger.addHandler(console_handler)
logger.addHandler(file_handler)

# 5. 输出日志（测试不同级别）
logger.debug("调试信息：变量 x = 10")  # 仅文件输出
logger.info("普通信息：用户登录成功")    # 控制台 + 文件输出
logger.warning("警告信息：参数格式不规范")  # 控制台 + 文件输出
logger.error("错误信息：数据写入失败")    # 控制台 + 文件输出
logger.critical("严重错误：系统即将崩溃")  # 控制台 + 文件输出
```

**输出效果**：
- 控制台（仅 INFO 及以上，简洁格式）：
  ```
  INFO - 普通信息：用户登录成功
  WARNING - 警告信息：参数格式不规范
  ERROR - 错误信息：数据写入失败
  CRITICAL - 严重错误：系统即将崩溃
  ```
- 文件 `app.log`（DEBUG 及以上，详细格式）：
  ```
  2025-11-20 15:40:00 - my_app - DEBUG - __main__:32 - 调试信息：变量 x = 10
  2025-11-20 15:40:00 - my_app - INFO - __main__:33 - 普通信息：用户登录成功
  2025-11-20 15:40:00 - my_app - WARNING - __main__:34 - 警告信息：参数格式不规范
  2025-11-20 15:40:00 - my_app - ERROR - __main__:35 - 错误信息：数据写入失败
  2025-11-20 15:40:00 - my_app - CRITICAL - __main__:36 - 严重错误：系统即将崩溃
  ```

### 3. 日志轮转：自动切割大日志文件
当日志文件过大时，`RotatingFileHandler`（按大小轮转）和 `TimedRotatingFileHandler`（按时间轮转）可自动切割文件，避免单个文件过大：

#### 示例 1：按大小轮转（`RotatingFileHandler`）
```python
import logging
from logging.handlers import RotatingFileHandler

# 创建 Logger
logger = logging.getLogger("size_rotating_log")
logger.setLevel(logging.INFO)

# 创建按大小轮转的 Handler
# maxBytes：单个文件最大大小（单位：字节，1024*1024=1MB）
# backupCount：保留历史文件数（超过则删除最旧的）
file_handler = RotatingFileHandler(
    "app_size.log",
    maxBytes=1024*1024,  # 1MB 轮转
    backupCount=5,       # 保留 5 个历史文件
    encoding="utf-8"
)
file_handler.setFormatter(logging.Formatter("%(asctime)s - %(levelname)s - %(message)s"))
logger.addHandler(file_handler)

# 模拟大量日志输出（触发轮转）
for i in range(10000):
    logger.info(f"测试按大小轮转日志 - 第 {i} 条")
```

**效果**：生成 `app_size.log`（当前日志）和 `app_size.log.1`~`app_size.log.5`（历史轮转文件），单个文件不超过 1MB。

#### 示例 2：按时间轮转（`TimedRotatingFileHandler`）
```python
import logging
from logging.handlers import TimedRotatingFileHandler

# 创建 Logger
logger = logging.getLogger("time_rotating_log")
logger.setLevel(logging.INFO)

# 创建按时间轮转的 Handler
# when：轮转时间单位（S：秒，M：分，H：时，D：天，W0-W6：周，midnight：午夜）
# interval：间隔时间（如 when='H'，interval=1 表示每小时轮转）
# backupCount：保留历史文件数
file_handler = TimedRotatingFileHandler(
    "app_time.log",
    when="H",          # 每小时轮转
    interval=1,        # 间隔 1 小时
    backupCount=24,    # 保留 24 小时历史文件
    encoding="utf-8",
    utc=True           # 使用 UTC 时间（默认本地时间）
)
file_handler.setFormatter(logging.Formatter("%(asctime)s - %(levelname)s - %(message)s"))
logger.addHandler(file_handler)

# 模拟日志输出
logger.info("按时间轮转日志 - 测试信息")
```

**效果**：每小时生成一个新文件（如 `app_time.log.2025-11-20_15`），保留最近 24 小时的日志文件。

### 4. 异常日志：记录堆栈信息
使用 `logger.error(exc_info=True)` 或 `logger.exception()` 可自动记录异常堆栈信息，便于定位问题：

```python
import logging

# 基础配置
logging.basicConfig(
    level=logging.ERROR,
    format="%(asctime)s - %(levelname)s - %(module)s:%(lineno)d - %(message)s",
    filename="error.log",
    encoding="utf-8"
)

logger = logging.getLogger("exception_log")

def divide(a, b):
    try:
        return a / b
    except ZeroDivisionError as e:
        # 方式 1：使用 exc_info=True 记录堆栈
        logger.error(f"除法错误：a={a}, b={b}", exc_info=True)
        # 方式 2：使用 exception() 自动包含堆栈（等价于 error(..., exc_info=True)）
        # logger.exception(f"除法错误：a={a}, b={b}")

# 触发异常
divide(10, 0)
```

`error.log` 文件内容（包含完整堆栈）：
```
2025-11-20 15:50:00,123 - ERROR - __main__:17 - 除法错误：a=10, b=0
Traceback (most recent call last):
  File "__main__.py", line 14, in divide
    return a / b
ZeroDivisionError: division by zero
```

### 5. 多模块日志：统一配置
大型项目多模块协作时，通过 `logging.getLogger(__name__)` 创建模块级 Logger，可继承全局配置，统一日志风格：

#### 项目结构
```
my_project/
├── main.py
└── utils.py
```

#### `utils.py`（子模块）
```python
import logging

# 创建模块级 Logger（名称为模块名 __name__）
logger = logging.getLogger(__name__)

def add(a, b):
    logger.debug(f"执行加法：a={a}, b={b}")
    result = a + b
    logger.info(f"加法结果：{result}")
    return result
```

#### `main.py`（主模块，配置全局日志）
```python
import logging
from logging.handlers import StreamHandler
import utils

# 全局日志配置（所有模块共享）
def setup_logging():
    # 创建 root Logger 并设置级别
    root_logger = logging.getLogger()
    root_logger.setLevel(logging.DEBUG)

    # 创建控制台 Handler
    handler = StreamHandler()
    handler.setFormatter(logging.Formatter(
        "%(asctime)s - %(name)s - %(levelname)s - %(message)s"
    ))
    root_logger.addHandler(handler)

if __name__ == "__main__":
    setup_logging()
    # 调用子模块函数，触发子模块日志
    utils.add(10, 20)
```

**输出结果**（主模块和子模块日志统一格式）：
```
2025-11-20 16:00:00,123 - utils - DEBUG - 执行加法：a=10, b=20
2025-11-20 16:00:00,124 - utils - INFO - 加法结果：30
```

### 6. 配置文件：分离日志配置（推荐）
将日志配置写入文件（如 `logging.conf`），与代码分离，便于维护和环境切换（开发/测试/生产）：

#### 示例：INI 格式配置文件（`logging.conf`）
```ini
[loggers]
keys=root,my_app

[logger_root]
level=DEBUG
handlers=console,file

[logger_my_app]
level=DEBUG
handlers=console,rotating_file
qualname=my_app
propagate=0  # 不向父 Logger 传播（避免重复输出）

[handlers]
keys=console,file,rotating_file

[handler_console]
class=StreamHandler
level=INFO
formatter=console_fmt
args=(sys.stdout,)

[handler_file]
class=FileHandler
level=DEBUG
formatter=file_fmt
args=("app.log", "a", "utf-8")

[handler_rotating_file]
class=logging.handlers.RotatingFileHandler
level=DEBUG
formatter=file_fmt
args=("app_rot.log", 1024*1024, 5, "utf-8")  # 1MB 轮转，保留 5 个

[formatters]
keys=console_fmt,file_fmt

[formatter_console_fmt]
format=%(levelname)s - %(message)s
datefmt=%Y-%m-%d %H:%M:%S

[formatter_file_fmt]
format=%(asctime)s - %(name)s - %(levelname)s - %(module)s:%(lineno)d - %(message)s
datefmt=%Y-%m-%d %H:%M:%S
```

#### 代码中加载配置文件
```python
import logging
import logging.config

# 加载配置文件
logging.config.fileConfig("logging.conf")

# 获取配置好的 Logger
logger = logging.getLogger("my_app")

# 输出日志
logger.debug("调试信息")
logger.info("普通信息")
logger.warning("警告信息")
```

### 7. 字典配置：更灵活的动态配置
Python 3.2+ 支持字典格式配置，适合动态生成配置（如从环境变量、配置中心获取参数）：

```python
import logging
import logging.config

# 日志配置字典
LOGGING_CONFIG = {
    "version": 1,
    "disable_existing_loggers": False,  # 不禁用已存在的 Logger
    "formatters": {
        "console": {
            "format": "%(levelname)s - %(message)s",
            "datefmt": "%Y-%m-%d %H:%M:%S"
        },
        "file": {
```

## 六、常见问题与避坑指南
### 1. 坑 1：`basicConfig()` 不生效
- **问题**：调用 `basicConfig()` 后，日志仍按默认配置输出（如未写入文件、格式不对）；
- **原因**：`basicConfig()` 仅在 **第一次创建 Handler 时生效**，若之前已创建 Logger 或 Handler（如其他模块导入时初始化了日志），则配置无效；
- **解决方案**：
  - 确保 `basicConfig()` 是程序中第一个日志相关调用；
  - 手动清除已存在的 Handler 后重新配置：
    ```python
    import logging

    # 清除已存在的 Handler
    root_logger = logging.getLogger()
    root_logger.handlers.clear()

    # 重新调用 basicConfig()
    logging.basicConfig(filename="app.log", level=logging.INFO)
    ```

### 2. 坑 2：日志重复输出
- **问题**：同一日志被多次打印（如控制台输出 2 次，或文件+控制台输出多次）；
- **原因**：
  1. 多个 Handler 被重复添加到 Logger；
  2. 子 Logger 的 `propagate=True`（默认），日志向父 Logger 传播，父 Logger 又有自己的 Handler；
- **解决方案**：
  - 避免重复调用 `addHandler()`（可先检查 Handler 是否已存在）；
  - 子 Logger 设置 `propagate=False`，禁止向父 Logger 传播；
  - 初始化 Logger 时清除已有 Handler：
    ```python
    logger = logging.getLogger("my_app")
    logger.handlers.clear()  # 清除已有 Handler
    logger.addHandler(console_handler)
    ```

### 3. 坑 3：日志级别设置无效
- **问题**：设置 `level=logging.DEBUG` 后，仍看不到 DEBUG 级日志；
- **原因**：Logger 和 Handler 都有级别阈值，**日志级别需同时满足 Logger 级别 <= 日志级别 <= Handler 级别** 才会输出；
- **解决方案**：确保 Logger 和所有关联的 Handler 级别都低于或等于目标日志级别：
  ```python
  logger = logging.getLogger()
  logger.setLevel(logging.DEBUG)  # Logger 级别设为 DEBUG

  handler = logging.StreamHandler()
  handler.setLevel(logging.DEBUG)  # Handler 级别也设为 DEBUG（关键）
  logger.addHandler(handler)
  ```

### 4. 坑 4：中文日志乱码
- **问题**：日志文件中中文显示为乱码（如 `???` 或特殊字符）；
- **原因**：文件 Handler 未指定 `encoding="utf-8"`，默认使用系统编码（如 Windows 下的 GBK）；
- **解决方案**：创建 FileHandler 时显式指定编码：
  ```python
  logging.FileHandler("app.log", encoding="utf-8")
  RotatingFileHandler("app.log", encoding="utf-8")
  ```

### 5. 坑 5：日志轮转失败（文件未切割）
- **问题**：配置了 `RotatingFileHandler` 但日志文件未按预期轮转；
- **原因**：
  1. `maxBytes`/`interval` 参数设置错误（如 `maxBytes=1024` 是 1KB，而非 1MB）；
  2. 日志文件路径无写入权限，或目录不存在；
- **解决方案**：
  - 检查参数单位（1MB=1024*1024 字节）；
  - 确保日志目录存在且有写入权限：
    ```python
    import os
    log_dir = "logs"
    if not os.path.exists(log_dir):
        os.makedirs(log_dir)  # 创建目录
    ```

### 6. 坑 6：异常日志未记录堆栈信息
- **问题**：`logger.error("错误信息")` 仅输出错误描述，无堆栈信息；
- **原因**：未启用 `exc_info=True`，默认不记录堆栈；
- **解决方案**：
  - 使用 `logger.error(exc_info=True)` 或 `logger.exception()`：
    ```python
    try:
        1 / 0
    except Exception:
        logger.exception("除法错误")  # 自动记录堆栈
        # 或
        # logger.error("除法错误", exc_info=True)
    ```

### 7. 坑 7：多进程日志错乱
- **问题**：多进程环境下，多个进程同时写入同一个日志文件，导致日志内容错乱（如行重叠、字符缺失）；
- **原因**：`FileHandler` 不支持多进程安全写入，多个进程同时操作文件会导致竞争条件；
- **解决方案**：
  - 使用 `MultiprocessingLogHandler`（第三方库 `logging-multiprocessing`）；
  - 每个进程使用独立的日志文件（如文件名包含进程 ID）；
  - 采用集中式日志收集（如通过网络输出到 ELK，避免直接写入本地文件）。

## 八、总结
logging 模块是 Python 日志记录的“标准解决方案”，核心价值在于 **分级可控、灵活配置、多场景适配**，从简单脚本到大型分布式应用都能满足需求。

### 学习路径建议
1. 基础入门：掌握 `basicConfig()` 快速配置，理解日志级别和基本格式；
2. 进阶应用：手动创建 Logger/Handler/Formatter，实现多目的地输出和日志轮转；
3. 实战优化：使用配置文件/字典配置，分离日志逻辑与业务代码；
4. 问题排查：熟悉常见坑点（重复输出、级别无效、乱码），掌握解决方案；
5. 高级扩展：结合日志收集工具（ELK）、自定义 Handler/Filter，适配复杂生产环境。

### 核心推荐
- 项目开发：优先使用 `logging.getLogger(__name__)` 创建模块级 Logger，避免使用 root Logger；
- 配置方式：小型项目用 `basicConfig()`，中大型项目用配置文件或字典配置；
- 生产环境：开启日志轮转、记录详细上下文（进程 ID、线程名、行号）、分离错误日志和普通日志；
- 异常记录：始终使用 `logger.exception()` 或 `exc_info=True` 记录异常堆栈，便于问题回溯。

通过熟练掌握 logging 模块，你可以为项目构建可靠、高效的日志系统，大幅提升调试效率和生产环境问题定位能力！
